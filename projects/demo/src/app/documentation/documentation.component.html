Documentation

@softwarity/store lets you persist UI preferences (visible columns, sort order, page size, filters, sidebar state…) directly in the browser — no server-side persistence needed. No API endpoints, no database tables, no extra round-trips: everything stays client-side, where UI preferences belong.

A single decorator or function call is all it takes. Annotate a property — or wrap an object with localStored() / sessionStored() — and every mutation (including nested properties and array methods) is automatically persisted to browser storage and restored on reload. No manual getItem / setItem, no boilerplate serialization logic — @softwarity/store handles it all for you.

User-scoped storage

Browser storage is shared across all users on the same browser profile. On shared workstations or kiosk machines, multiple people may log in to the same app one after another. Without scoping, one user's persisted preferences (columns, filters, sort order…) would leak to the next user.

By providing a userId signal, every storage key is automatically prefixed (userId_storageKey). When the user changes (login / logout / switch), the library reloads each stored object from the new user's namespace — preferences follow the user, not the machine.

@LocalStored / @SessionStored — Decorator API

Annotate class fields for automatic persistence with deep mutation tracking. Nested property changes and array methods trigger saves automatically.

Deep mutation tracking

The decorator wraps stored objects with native Proxy. All nested mutations, array methods, direct index assignments, and new properties trigger automatic saves.

Schema version management

When the schema changes, increment the schema version. Old stored data is discarded and replaced with the new defaults.

Only @LocalStored and localStored() support schema versioning. @SessionStored and sessionStored() do not need it (data is short-lived).

Stored API — localStored / sessionStored

Same deep mutation tracking as decorators, but without annotations. Returns a StoredSignal<T> with plain property access, deep tracking on nested objects and arrays, and $prop() reactive signals at every nesting level for templates.

Features

  • Decorator API — @LocalStored / @SessionStored with deep mutation tracking on nested objects and arrays, plus $prop() reactive signals at every nesting level
  • Stored API — localStored() / sessionStored() — same deep mutation tracking and $prop() signals as decorators, but without annotations
  • Schema versioning — Increment the schema version to discard stale data when the schema changes
  • User-scoped — Prefix storage keys with a user ID via provideStore({{ '{' }} userId: () => mySignal {{ '}' }})
  • Cross-tab sync — onStorageChange() for localStorage changes from other tabs
  • Error resilient — Handles corrupted JSON and QuotaExceededError gracefully
  • SSR compatible — FakeStorage fallback when window is undefined
  • Zoneless — No dependency on zone.js
